# argo-git-flow

ローカルの変更を Git の作業フローに沿って安全に進めるために使う。

## 重視すること

- ブランチを変更する前にリポジトリ状態を確認する
- ファイル変更を伴う作業前に、原則として新しい `claude/` ブランチを発行する
- ブランチ命名、ステージング、コミット、プッシュ、`gh` による PR 作成を明示的に進める
- 関係ないファイル、秘密情報、実行時データ、ユーザーの別作業を誤ってステージングしない
- 破壊的操作や履歴を書き換える操作は、事前確認なしに実行しない
- PR 作成や push で終わらせず、CI がある場合は全チェック通過まで見届ける
- 変更、コミット、プッシュ、未対応事項を具体的に報告する

## 手順

1. `git status --short --branch` で現在の状態を確認する。
2. ファイル変更を伴う作業では、原則として新しい作業ブランチを発行する。
3. 作業ブランチは `claude/add-login-test` のように `claude/` で始める。
4. worktree が clean なら `git switch -c <branch-name>` で作業ブランチを作成する。
5. 未コミット変更がある場合は、混入や喪失を避けるため、コミット、stash、現ブランチ維持のどれにするか確認する。
6. 既存ブランチへ移動する場合は、ローカル変更を失わないことを確認してから `git switch <branch-name>` を使う。
7. ステージング前に `git diff`、必要に応じて `git diff -- <path>` で差分を確認する。
8. `git add <path>` で意図したファイルだけをステージングする。
9. コミット前に `git diff --cached` で staged diff を確認する。
10. staged diff に合う簡潔なメッセージでコミットする。
11. 新しいブランチは `git push -u origin <branch-name>`、upstream 設定済みなら `git push` でプッシュする。
12. プッシュ済みブランチと base branch を確認してから `gh pr create` で PR を作成する。
13. PR 作成後（および PR があるブランチへの push 後）、CI が設定されていれば「CI 自己修正ループ」で全チェック通過まで確認する。

## ブランチ命名

- Claude が作る新しい作業ブランチは `claude/` で始める。
- `claude/update-readme` や `claude/fix-login-test` のように、prefix の後ろは短い lowercase hyphen 形式にする。
- ユーザーが `claude/` なしのブランチ名を指定した場合、正確な名前を使うよう明示されていなければ `claude/` を付ける。
- 既存のユーザーブランチは、ユーザーが依頼しない限りリネームしない。

## ブランチ安全ルール

- `git reset --hard`、`git checkout --`、`git clean`、force push、rebase、ブランチ削除は、明示確認なしに実行しない。
- 未コミット変更が失われたり別ブランチへ混ざったりする可能性がある場合は、ブランチを切り替えない。
- dirty worktree の状態でブランチを切り替える必要がある場合は、対象ファイルを要約し、コミット、stash、現ブランチ維持のどれにするか確認する。
- ブランチ操作には、原則として古い `git checkout` ではなく `git switch` を使う。

## ステージングとコミットの安全ルール

- 原則としてファイルパスを指定してステージングする。
- `git add .` は、ユーザーが現在の全変更を含めたいと明示し、差分確認済みの場合だけ使う。
- コミット前に `git diff --cached` で staged diff を確認する。
- 想定外のファイルが出てきた場合は、含める前に止まって確認する。
- 秘密情報、ローカル認証情報、実行時データ、マシン固有ファイル、大きな生成物は、ユーザーが明示し、かつリポジトリに適切な場合以外は追加しない。

## PR 作成

- PR 作成前に、ブランチ名、base branch、push 先 remote を確認する。
- 認証やリポジトリアクセスが不明な場合は `gh auth status` を確認する。
- 重複 PR を避けるため、`gh pr status` または `gh pr list --head <branch-name>` で既存 PR を確認する。
- PR は `gh pr create --base <base-branch> --head <branch-name> --title <title> --body <body>` で作成する。
- レビュー準備ができていない場合は `--draft` を使う。
- PR 本文には短い変更概要と検証結果を含める。
- ローカル環境で `gh pr create` を実行できない場合は、正確なブランチ名と簡潔な PR タイトル、本文案を提示する。

## CI 自己修正ループ

PR 作成や push で作業を終わらせず、CI の結果まで見届けて自分で修正する。

手順:

1. `gh pr checks <PR番号> --watch` で CI の完了を待つ（PR が無い場合は `gh run list --branch <branch-name>` で該当 run を特定する）。
2. 全チェック pass なら結果を報告して終了する。
3. 失敗があれば `gh run view <run-id> --log-failed` で失敗ログを取得し、原因を特定する。
4. 修正の前に、ローカルで同じチェックを再現する（そのリポジトリの品質ゲート・テスト・リンタを実行する）。ローカルにツールが無ければ導入してから直す。CI でしか再現しない場合はログを根拠に修正する。
5. 修正をローカルで検証してから、意図したファイルだけをステージングし、原因が分かるメッセージでコミットして push する。
6. 1 に戻る。修正の試行は最大 5 回まで。5 回で通らない場合は停止し、原因分析と試行の経緯を報告してユーザーの判断を仰ぐ。

ルール:

- CI を通すこと自体を目的に、テストの削除・skip 追加・アサーション緩和・CI 設定の無効化をしない。それが正しい修正だと判断した場合は、理由を明示してユーザーに確認してから行う。
- force push や履歴書き換えで CI をやり直さない。修正は新しいコミットで積む。
- 同一コードで成否が揺れる（flaky と思われる）場合、再実行は 1 回まで。それでも不安定なら flaky と判断した根拠を添えて報告し、修正ループには数えない。
- 各イテレーションで、失敗原因・修正内容・結果を 1〜2 行で報告する。
- 修正が別の関心事に及ぶ場合（依存の更新、CI 設定自体の不具合など）は、ループ内で抱え込まずユーザーに確認する。

## 出力

- 現在のブランチと対象ブランチ
- ステージング、コミットしたファイル
- コミットを作成した場合は commit hash
- push 結果と remote branch
- PR URL または PR 文案
- CI の最終結果と自己修正ループの実行回数（CI がある場合）
- 実行しなかった手順、ブロックされた操作、必要なユーザー確認
