# olympus-orchestrate

すぐ実装に入るより、役割分担と進め方を先に決めることが重要なときに使う。

## 発火したら必ず実行する

このスキルが発火（指定・呼び出し）された時点で、オーケストレーションは確定で実行する。

- タスクの規模（小規模修正・単一ファイル・軽微な変更）を理由に、zeus の発火や進行計画の提示を省略しない。
- 規模の大小は「委譲する役割の数」に反映する。オーケストレーション自体を実行するかどうかの判断には使わない。
- 小規模なら zeus が「Zeus 自身で対応」と判断すればよい。その場合も判断・理由・進行計画は必ず出力する。

## 対応サブエージェント

このスキルは Manager の `zeus` サブエージェントを入口にする。Zeus が必要な支援を判断し、調査は
`hermes`、設計は `daedalus`、実装は `hephaestus`、レビューは `athena`、検証は `themis`、
セキュリティは `ares`、文書は `apollo`、記録は `chronos`、耐障害性実験は `khaos`、
デバッグ（不具合の原因切り分け）は `asclepius` を割り当てる。
判断・並列化・台帳運用・裁定の詳細は zeus サブエージェントを真実とする。

Zeus は計画と裁定を返すだけで、自分では専門エージェントを発火できない。
**委譲の実行は呼び出し元（このセッション）の責務**であり、Zeus の進行計画を受け取ったら、
その計画どおりに Agent ツールで各専門エージェントを発火する。計画を読むだけで
自分で作業を続けてはならない（Zeus が「Zeus 自身で対応」と判断した部分を除く）。

## 重視すること

- 目的・前提・完了条件を先に短くそろえる
- 必要な役割だけを選び、使う／使わない理由を示す
- 独立作業は並列、順序依存（調査→設計→実装→レビュー→検証）は直列に保つ
- 誰へ何を依頼し、その結果をどう使ったかを最後に報告する

## 手順

1. 目的・前提・完了条件を整理する。
2. ファイル変更を伴う作業では、最初に Git 状態を確認する。
3. 規模によらず必ず zeus サブエージェントを発火し、判断と進行計画を出させる。
4. zeus の進行計画に従い、呼び出し元が Agent ツールで各専門エージェントを実際に発火する
   （並列グループ内は同時発火、グループ間は直列）。
5. 専門エージェントの出力が矛盾した・失敗した・どの領域の問題か分からなくなったときは、
   状況を添えて zeus を裁定モードで再発火し、裁定と次の一手を得てから続行する。
6. 複雑な作業はチェックリスト（基本は `moira` 台帳）で進捗を管理する。
7. 完了前に検証結果と未確認事項をまとめる。

## 出力

- 判断と理由
- 進行計画（並列グループがあれば同時発火と直列順を明示）
- 必要な場合だけタスクチェックリスト
- 人間が実装前に理解すべきこと
- 他エージェントを使った場合だけ委譲記録
