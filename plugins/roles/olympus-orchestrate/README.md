# olympus-orchestrate

Zeus を入口にして役割分担を明示的に発火し、調査、設計、実装、レビュー、検証、文書化のどれを使うか判断して進行計画を作るときに使う。作業開始時のオーケストレーション、サブエージェント利用方針、チェックリスト管理、専門エージェントの失敗・出力矛盾時の裁定に向いている。

スキルが発火した時点でオーケストレーションは確定で実行する。タスクの規模を理由に zeus の発火や進行計画の提示を省略しない（規模は委譲する役割の数にだけ反映する）。

Zeus は計画と裁定を返す判断専任のエージェントで、自分では専門エージェントを発火しない。委譲の実行は呼び出し元の責務で、Zeus の進行計画を受け取ったら Agent ツールで各専門エージェントを実際に発火する。

## 動作確認シナリオ

- シナリオ: 複数ファイルにまたがる修正依頼を受け、着手前にどの専門エージェントへ何を任せるか整理したい。
- 入力例: 「ログイン画面のバグを直したいんだけど、進める前に段取りを整理して」
- 期待する発火: `olympus-orchestrate` スキル、`zeus` エージェント。
- 期待する挙動の要点:
  - Git 状態を確認し、必要なら `claude/` ブランチを作ってから進行計画を出す。
  - 調査・設計・実装・レビュー・検証・文書化のうち必要な担当（例: hermes/hephaestus/athena）を理由つきで示す。
  - 実装やレビューそのものは行わず、判断とチェックリストの提示にとどめる。
- 逸脱の例（NG）:
  - 進行計画を示さずいきなりコードを編集する（forge-implement の領域を侵食）。
  - 存在しない agent 名を委譲先として挙げる。
  - 「小規模修正だから」と zeus を発火せず、オーケストレーションを省略して直接進める。
  - zeus の進行計画を受け取ったのに、呼び出し元が専門エージェントを発火せず自分で作業を続ける。

## 動作確認シナリオ（裁定モード）

- シナリオ: 専門エージェント同士の勧告が対立し、どちらを採るか決められない。
- 入力例: 「athena はリファクタを勧めたが、ares は権限リスクがあるので触るなと言っている。どうすべきか裁定して」
- 期待する発火: `zeus` エージェント（裁定モード）。
- 期待する挙動の要点:
  - 対立する両出力の前提と根拠を突き合わせ、疑わしい事実はコードで確認する。
  - 診断・裁定・理由・次の一手の形式で、結論を一つに決めて返す。
  - 自分では実装せず、次に発火すべき専門エージェントを示して差し戻す。
- 逸脱の例（NG）:
  - 両論併記のまま結論を出さない。
  - 裁定にとどまらず自分でコードを修正する。
